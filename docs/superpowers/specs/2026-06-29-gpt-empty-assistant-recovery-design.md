# GPT 空 assistant 自动恢复设计

## 背景

GPT 页面偶尔会在问题提交后进入异常状态：assistant 节点不存在，或节点存在但没有文字和生成图片。此时刷新页面通常可以恢复。

## 目标

GPT 问题确认提交后仅执行一次延迟检查：等待 3 秒，若 assistant 仍为空，则根据页面 URL 是否变化选择恢复方式。

- URL 已变化：刷新页面，刷新后只恢复响应监听，不重复发送问题。
- URL 未变化：刷新页面，刷新后重新上传必要附件并重新发送问题。

Gemini 流程不变。

## 判定规则

发送前记录当前 URL。问题确认提交成功后启动一次性 3 秒延迟，不做周期轮询。

assistant 满足以下任一条件时视为空：

- 当前任务对应的 assistant 节点不存在。
- assistant 节点存在，但没有非空文字，也没有已识别的生成图片。

assistant 非空时不执行恢复，继续现有监听流程。

## 组件与数据流

内容脚本负责延迟检查和判定，后台脚本负责刷新与恢复：

1. GPT 发送前，内容脚本保存页面 URL。
2. GPT 提交成功后，内容脚本启动正常监听，同时安排一次 3 秒后的检查。
3. 检查发现 assistant 非空时结束，不发送恢复消息。
4. 检查发现 assistant 为空时，内容脚本向后台上报恢复请求，并明确恢复模式：
   - `monitor_only`：URL 已变化。
   - `resubmit`：URL 未变化。
5. 后台沿用现有 `stalled`、`refreshing`、刷新次数限制和标签页重载机制。
6. 页面重载完成后，后台通过 `START_JOB` 携带恢复模式：
   - `monitor_only` 直接进入 `monitorJob`。
   - `resubmit` 执行正常发送流程，包括必要的附件上传。

恢复模式必须由触发检查时的 URL 比较结果确定，不依赖刷新后的 DOM 状态推断。

## 状态与错误处理

- 两种恢复都增加现有 `refreshCount`，并受 `maxRefreshPerJob` 限制。
- 达到最大刷新次数时沿用现有 `needs_manual` 处理。
- 刷新或重新连接内容脚本失败时沿用现有错误上报。
- 一次性检查必须响应任务的 `AbortSignal`；任务终止或被替换后不得触发旧任务恢复。
- 恢复请求发出后，当前内容脚本停止该任务监听，避免刷新期间继续上报状态。

## 测试

使用可单元测试的纯判定函数覆盖：

- 3 秒前不检查，3 秒后只检查一次。
- 非 GPT 任务不触发。
- assistant 不存在时视为空。
- assistant 空节点时视为空。
- assistant 有文字或生成图片时不触发。
- assistant 为空且 URL 已变化时选择 `monitor_only`。
- assistant 为空且 URL 未变化时选择 `resubmit`。

恢复消息和后台处理还需覆盖：

- `monitor_only` 刷新后不调用上传或发送流程。
- `resubmit` 刷新后重新执行正常发送流程。
- 两种模式均复用刷新次数上限。

完成后运行 `npm run build`、`npm run check` 和 `npm test`。
